# 用户体验和页面流程

## 信息架构

当前 MVP 使用 4 个主视图：

- Home: 宠物、今日浪费阻断决策、临期提醒。
- Inventory: 所有库存、筛选、编辑。
- Check & Add: 购物清单检查、小票上传、照片上传、手动添加。
- Results: 减少浪费成果和宠物成长记录。

`Plan` 目前合并在 Dashboard 中：规则引擎生成 Today missions，AI daily plan 在 Dashboard 内生成菜谱步骤和可提交使用任务。

详细前端视觉和互动规范见 [FRONTEND_EXPERIENCE_DESIGN.md](FRONTEND_EXPERIENCE_DESIGN.md)。

## 首页 Dashboard

首页需要第一屏直接呈现：

- 宠物房间和宠物当前状态。
- 今日最该处理的 1 到 3 个浪费阻断卡片。
- 一个主行动按钮：检查购物清单。
- 一个次行动按钮：查看库存。
- 风险摘要：今天到期、本周到期、已过建议日期。

首页不应做成营销落地页。它应直接是可使用的应用界面。

首页体验目标：

- 桌面端像“冰箱任务中心 + 宠物房间”。
- 移动端顶部显示宠物，中间显示今日任务，底部显示导航。
- 用户 5 秒内能理解：今天哪件食物可能浪费、为什么、默认应该拦截购买还是救援食物、处理后宠物和 impact 为什么变化。

## 任务公布

今日计划在 UI 中以 `Mission` 展示，而不是普通列表。

同步规则：

- Today missions 是权威任务来源，由规则引擎从 `active` 食物生成。
- AI daily plan 是临时执行层，基于当前 Today missions 和 active 食物生成菜谱与 usage tasks。
- 如果用户在 Today mission 中把 AI plan 正在使用的食物标记为 frozen/used/shared/discarded，AI plan 必须自动移除相关 task，并提示计划已同步。
- Frozen 食物不进入 Today missions，也不应显示 `use soon`；Inventory 中显示为 `frozen pause`。
- `checked` 不会移除 Today mission。它只表示用户完成了第一步检查，下一次重算必须显示第二步决策。
- `check` 和 `freeze` 不能作为同一张卡的同步按钮出现。低置信、过建议日期、或临期 meat/seafood/prepared food 先检查；检查后再给 use/freeze/share/discard。

任务类型：

- `rescue`: 使用临期物品。
- `freeze`: 冷冻以延长可用窗口。
- `share`: 分享或捐赠。
- `review`: 确认日期或识别结果。
- `cleanup`: 记录已丢弃或不安全物品。

每张任务卡必须包含：

- 任务标题。
- 关联物品。
- 推荐原因。
- 建议行动。
- 奖励预览。
- 阶段标签：`Step 1: Check`、`Step 2: Decide` 或 `Waste blocker`。
- 主按钮。
- 次按钮。

示例：

```text
拯救菠菜
原因：它已经在冰箱里 5 天了，建议优先处理。
建议：加到鸡蛋、面条或三明治里。
奖励：宠物 +10 健康
按钮：我用掉了 / 我冷冻了 / 今天不想吃
```

## Check & Add 流程

页面分成两个清晰区块：

- `Before buying`: 用户先粘贴购物清单，系统只回答“哪些不该买、哪些要减少、哪些需要先看库存”。
- `Add food`: 用户购买后或整理冰箱时，再通过小票、照片或手动表单加入库存。

购物清单结果卡只展示三层信息：建议芯片、浪费原因、默认动作。避免同时出现多组“检查库存”按钮。

### 小票上传

1. 用户选择“上传小票”。
2. 页面提示隐私风险：小票可能包含门店、时间、会员或付款信息。
3. 用户上传图片。
4. 系统显示识别中状态。
5. 系统展示候选条目：
   - 原始识别文本；
   - 推断物品名称；
   - 类别；
   - 数量；
   - 单位；
   - 购买日期；
   - 建议使用日期；
   - 包装规格或识别备注；
   - 置信度。
6. 用户拒绝非食物条目，修正名称、类别、数量、单位、日期和备注。
7. 用户逐项确认，或批量确认所有未拒绝候选项。
8. 如果候选项与现有库存重复或造成高风险过量，宠物先显示购买防护提示。

### 照片上传

1. 用户选择“拍照/上传照片”，或把图片拖拽到 AI image recognition 上传区。
2. 用户上传冰箱、食品柜、一堆食物或单个物品照片。
3. 系统识别可见物品。
4. 用户逐项编辑、拒绝或确认。
5. 对无法判断建议使用日期的物品，系统要求用户选择：
   - 今天买的；
   - 最近一周买的；
   - 不确定；
   - 手动填日期。

提示文案应说明：冰箱照片适合识别可见物品，但可能漏掉隐藏、遮挡或堆叠的食物。

### 手动添加

最小字段：

- 名称
- 类别
- 数量
- 储存位置
- 购买日期
- 建议使用日期

可选字段：

- 单位
- 包装状态
- 备注
- 图片
- 条码

### 购买防护

当用户准备添加与 active 库存重复或数量明显过多的物品时，系统先显示宠物拦截提示，而不是直接加入库存。

触发场景：

- 名称与现有 active 物品相同或高度相似。
- 同类别高风险物品已有多个待处理。
- 同物品数量超过建议阈值。

提示必须包含：

- 宠物语气的阻止文案。
- 具体库存原因。
- 建议先处理的现有物品。
- 主按钮：先处理现有库存。
- 次按钮：仍然添加。

示例：

```text
Pet says: Hold on. We already have 6 bananas on the counter.
Try rescuing or freezing them before adding more.
```

宠物可以拒绝默认添加，但不能完全剥夺用户控制权。用户二次确认后仍可添加。

## 库存页面

库存列表应显示：

- 物品名
- 数量
- 储存位置
- 风险标签
- 建议使用日期
- 推荐行动

风险标签：

- `Use by today`
- `Use soon`
- `Stable`
- `Review`
- `Past suggested date`
- `Frozen pause`

筛选：

- 默认筛选为 `available`，表示 `active + frozen`。
- `active` 表示可进入今日任务和 AI daily plan。
- `frozen` 表示已保存但暂停计划。

操作：

- 使用
- 冷冻
- 分享/捐赠
- 丢弃
- 编辑
- 删除

Frozen 食物仍是库存，不是“用完”。它保留数量，储存位置应同步为 freezer，并从 urgent mission 中移除。

## 计划页面

计划页面分为：

- Today
- This week
- Backup ideas

每个计划项应包含：

- 推荐物品。
- 推荐原因。
- 建议行动。
- 预计耗时。
- 完成按钮。
- 换一个按钮。

示例：

```text
Use spinach today
Reason: leafy greens are high priority and this item was bought 5 days ago.
Action: add to eggs or noodles.
```

## 宠物反馈

宠物状态变化应即时但温和：

- 完成计划：宠物变开心、健康值上升。
- 冷冻临期食物：宠物获得能量。
- 分享/捐赠：宠物获得爱心。
- 丢弃未计划处理的物品：宠物低落。
- 连续多天无行动：宠物变困或饿。
- 点击宠物：显示当前提醒或鼓励。
- 完成任务：宠物做轻量动画，并显示奖励 toast。

避免文案：

- “你很浪费”
- “你失败了”
- “你害了宠物”

推荐文案：

- “今天还有 2 个物品需要关注。”
- “试试先处理菠菜，宠物会恢复精神。”
- “丢弃已记录。下次我们可以提前提醒。”
- “冰箱里有个东西快被遗忘了。”
- “干得好，我感觉精神多了。”
- “等一下，我们已经有香蕉了。先救它们再买新的？”

## 空状态

### 新用户

- 显示宠物孵化状态。
- 给出 3 个入口：上传小票、上传照片、手动添加。
- 提供演示数据按钮，方便 hackathon 展示。

### 无临期物品

- 宠物健康。
- 展示“本周库存稳定”。
- 推荐整理库存或添加下一次购物。

### 识别失败

- 显示失败原因。
- 保留上传图片预览。
- 提供手动添加入口。
- 不把失败解释成用户错误。

## 可访问性

- 所有按钮有文本或 aria-label。
- 风险状态不能只靠颜色表达。
- 图片识别结果要有文本列表。
- 宠物动画不能是唯一反馈通道。
- 移动端单手操作优先，主要按钮靠近底部。
